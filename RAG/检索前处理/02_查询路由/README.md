# 02 查询路由学习手册（详细版）

## 1. 先建立一个正确认知

查询路由（Query Routing）不是“把 query 随便分发”，而是决定：

1. 这个查询应该走哪种检索器（dense/sparse/hybrid/multimodal）。
2. 这个查询是否需要先做翻译（重写、分解、澄清）再检索。
3. 这个查询是否要走特定索引（按领域、按时间、按权限）。

如果路由错了，后续检索和生成都很难补救。

## 2. 两种核心路由

### 2.1 逻辑路由（Rule-based）

定义：基于显式规则做路由决策。

适用场景：

1. 业务模式稳定（例如型号查询、时间过滤、图片搜索）。
2. 你要求高可解释性与可控性。
3. 需要先快速上线一个稳定版本。

优点：

1. 可解释（每条路由都能说清规则原因）。
2. 可控（改规则立即生效）。

缺点：

1. 泛化能力弱。
2. 规则会不断膨胀。

### 2.2 语义路由（Semantic）

定义：利用模型理解 query 意图，再选择路由。

常见做法：

1. LLM 分类路由（让模型返回 route + reason + confidence）。
2. 向量原型路由（query 向量与路由原型向量做相似度）。

优点：

1. 泛化强，能处理表达变体。

缺点：

1. 误判不可避免，需要评测和兜底。
2. 成本和延迟通常高于纯规则。

## 3. 推荐架构：规则兜底 + 语义细分

推荐执行顺序：

1. 先逻辑路由做高置信规则匹配（例如“图片/照片”）。
2. 未命中规则时再走语义路由。
3. 语义路由低置信度时回落到 hybrid 或默认 dense。

这叫“分层路由”，是工程上最稳的方案。

## 4. 路由目标（本章统一）

1. `dense_retriever`：语义问答、解释型问题。
2. `sparse_retriever`：型号、编号、术语精确匹配。
3. `hybrid_retriever`：对比、方案选型、复杂查询。
4. `multimodal_retriever`：图片、图像、视觉内容查询。

## 5. 本目录 demo 说明

1. `demos/demo_01_logical_routing.py`
- 学习重点：规则设计与可解释输出。

2. `demos/demo_02_semantic_routing.py`
- 学习重点：让千问返回结构化路由结果。

3. `demos/demo_03_combined_routing.py`
- 学习重点：规则优先 + 语义兜底 + 低置信度回退。

4. `demos/demo_04_routing_eval.py`
- 学习重点：批量样本评估路由准确率。

## 6. 路由评估指标

1. Route Accuracy：路由是否命中标注目标。
2. Confidence Calibration：置信度是否可信。
3. Downstream Gain：路由后检索指标是否提升。

## 7. 常见错误

1. 所有 query 只走一个 route（路由形同虚设）。
2. 语义路由无兜底（误判直接伤害检索）。
3. 只看单条案例，不做批量评估。

## 8. 学习节奏建议

1. 先运行 `demo_01` 看规则路由。
2. 再运行 `demo_02` 看语义路由。
3. 再运行 `demo_03` 理解组合决策。
4. 最后跑 `demo_04` 看指标和误判样本。
