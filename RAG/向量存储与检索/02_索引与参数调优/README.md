# 02 Qdrant 索引与参数调优学习手册（可执行版）

## 1. 本模块你真正要学会什么

1. 明确 `m`、`ef_construct`、`hnsw_ef` 分别影响什么。
2. 能用同一份数据做参数实验，而不是靠感觉调参。
3. 能输出“召回-延迟”权衡结论，并给出推荐参数。

## 2. 核心知识点与细节

1. `m`（图连接度）
- 更高 `m` 通常提高召回上限。
- 代价是更高内存占用和更慢建索引。

2. `ef_construct`（建索引搜索宽度）
- 越高通常索引质量越好。
- 代价是构建时间更长。

3. `hnsw_ef`（查询搜索宽度）
- 越高通常查询召回越高。
- 代价是单次查询延迟上升。

4. 参数关系直觉
- `m/ef_construct` 决定“索引天花板”。
- `hnsw_ef` 决定“查询时是否接近天花板”。

## 3. 如何实现参数实验（标准流程）

1. 固定 embedding 模型：千问 `text-embedding-v3`。
2. 固定实验数据：同一批文本、同一批 query、同一 ground truth。
3. 针对每组参数：
- 重建 collection（包含 HNSW 配置）。
- 写入同一数据。
- 跑同一查询集并统计 Recall@K 与延迟。
4. 最后输出对比表，选择满足目标的最优折中。

## 4. 本模块 Demo 说明

1. `demo_index_params.py` 提供完整实验：
- 自动生成样本数据并用千问向量化
- 运行多组 `m/ef_construct/hnsw_ef`
- 输出每组参数的 `Recall@3`、平均延迟、P95 延迟

2. 环境变量
- `DASHSCOPE_API_KEY`：千问密钥（必需）
- `QDRANT_API_KEY`：远程 Qdrant 鉴权（按你的实例要求）
- `QDRANT_URL`：默认 `http://115.190.125.94/`

## 5. 参数结果怎么解读

1. 如果 Recall 太低：
- 先增大 `hnsw_ef`。
- 再考虑提高 `m` 或 `ef_construct` 重建索引。

2. 如果延迟太高：
- 先降低 `hnsw_ef`。
- 再回看 TopK 是否过大、过滤条件是否过重。

3. 如果资源紧张：
- 保守设置 `m`，靠适中 `hnsw_ef` 找平衡点。

## 6. 常见坑

1. 一次改多个参数，导致无法归因。
2. 用不同数据集对比参数，结论不可信。
3. 只看平均延迟，不看 P95 长尾延迟。

## 7. 本模块验收标准

1. 你能跑出至少 4 组参数对比结果。
2. 你能解释某组参数为什么“更适合当前目标”。
3. 你能给出下一轮调参计划，而不是只报最优值。
