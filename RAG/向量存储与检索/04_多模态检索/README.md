# 04 Qdrant 多模态检索学习手册（文本 + 图像）

## 1. 你在这一模块要学会什么

1. 理解多模态检索不是“多存几个字段”，而是“多语义空间协同检索”。
2. 掌握 Qdrant Named Vectors 的数据建模方法。
3. 能独立实现三类检索：文搜图、图搜图、图搜文。
4. 能判断跨模态结果差是建模问题、向量问题还是检索参数问题。

## 2. 核心概念（先建立认知）

1. 多模态对象
- 一条业务对象（例如一张商品图）可以有多种表达：
- 文本表达：标题、描述、标签
- 图像表达：图像特征向量

2. Named Vectors
- 在 Qdrant 中，一个 point 可以挂多个向量字段。
- 例如：`text_vector`、`image_vector`
- 这样可以在同一 collection 内做不同模态的检索。

3. 跨模态检索
- 文搜图：输入文本，用图像向量空间查图。
- 图搜图：输入图像，用图像向量空间查相似图。
- 图搜文：输入图像，用文本向量空间查文字说明。

## 3. 为什么多模态检索难

1. 语义空间对齐问题
- 文本模型和图像模型如果不对齐，跨模态召回会不稳定。

2. 粒度不一致问题
- 图像强调视觉细节，文本可能只描述抽象概念。

3. 评估难题
- 跨模态正例定义更复杂，通常需要人工标注支持。

## 4. 工程实现方法（标准步骤）

1. 定义 point schema
- 主键：`id`（稳定唯一）
- payload：`object_id`、`title`、`category`、`tags`
- vectors：`text_vector`、`image_vector`

2. 构造两套向量
- 文本向量：对标题/描述做 embedding
- 图像向量：真实项目应使用图像 embedding 模型

3. 创建 Named Vectors collection
- 在 `vectors_config` 中为每个向量字段分别定义 `size + distance`

4. 写入 points
- 同一条 point 同时写入两个向量字段

5. 检索
- 文搜图：`using='image_vector'` + 文本 query 的“图像语义向量”
- 图搜图：`using='image_vector'` + 输入图像向量
- 图搜文：`using='text_vector'` + 输入图像语义向量

## 5. 本 Demo 的教学取舍（你要知道）

为了先把“多模态检索机制”讲清楚，本模块 demo 采用：

1. 文本向量：千问 embedding（真实）
2. 图像向量：用“图像描述文本”经千问 embedding 近似模拟（教学用）

这不是最终生产方案，但能完整演示：

1. Named Vectors 建模
2. 多模态查询路由（`using`）
3. 文搜图/图搜图/图搜文三类流程

后续你掌握机制后，再替换为真实图像 embedding 模型即可。

## 6. 参数与策略

1. 距离度量
- 文本、图像向量都可先用 Cosine 作为基线。

2. TopK
- 学习阶段先用 `TopK=3~5` 便于人工分析。

3. payload 过滤
- 多模态检索同样建议带业务过滤（例如 `category`）。

## 7. 常见错误与排查

1. 向量维度不一致
- 不同命名向量字段各自维度必须固定且匹配写入值。

2. using 字段写错
- 查询时 `using='image_vector'` 或 `using='text_vector'` 必须与 schema 一致。

3. 跨模态效果差
- 先检查语义对齐，再检查数据标注与 payload。

## 8. 本模块 Demo 学习任务

1. 先运行 `demo_multimodal_search.py` 看三类检索输出。
2. 修改 query 文本，观察文搜图排序变化。
3. 把 `category` 过滤加到检索里，观察结果可控性提升。
4. 调整 TopK 和 distance，记录结果差异。

## 9. 通过标准

1. 你能解释 Named Vectors 的设计理由。
2. 你能独立写出文搜图与图搜图查询。
3. 你能识别至少 2 个跨模态失败样本并给出改进方向。
