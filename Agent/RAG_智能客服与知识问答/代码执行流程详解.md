# RAG 智能客服项目代码执行流程详解（初学者版）

本文档面向初学者，按“程序怎么跑起来、请求怎么流转、每一步做了什么”来讲清楚这个项目。

## 1. 项目结构总览

你当前项目目录（核心部分）是：

- `app/main.py`：FastAPI 入口，定义 HTTP 接口。
- `app/chains.py`：LLM 路由与 RAG 检索链的核心逻辑。
- `app/config.py`：读取环境变量（Qdrant、千问模型等配置）。
- `app/embeddings.py`：自定义千问 Embedding 适配器。
- `app/router_schemas.py`：接口请求/响应的数据结构定义（Pydantic）。
- `app/memory.py`：会话记忆（当前为进程内内存）。
- `app/tools.py`：工单工具（当前为内存模拟数据库）。
- `ingest/loader.py`：加载知识库文本并切分 chunk。
- `ingest/index_qdrant.py`：把 chunk 向量化并写入 Qdrant。
- `data/*.txt`：知识库原始文本。

可以把它理解为两条主线：

1. 离线建库线（ingest）：`data -> 切分 -> embedding -> Qdrant`
2. 在线问答线（app）：`用户问题 -> 意图判断 -> FAQ检索回答 或 工单创建`

---

## 2. 先看离线建库流程（`ingest/index_qdrant.py`）

这是“把知识库准备好”的过程，通常先执行一次。

执行命令：

```bash
python ingest/index_qdrant.py
```

### 2.1 入口与路径处理

在 `index_qdrant.py` 顶部会做两件事：

1. 计算项目根目录（`PROJECT_ROOT`）
2. 把根目录加入 `sys.path`

目的：保证脚本直接运行时，也能正确 `import app.embeddings`。

### 2.2 读取配置

`load_dotenv()` 会从 `.env` 加载环境变量，然后读取：

- `QDRANT_URL`：Qdrant 地址
- `QDRANT_API_KEY`：Qdrant 密钥
- `QDRANT_COLLECTION`：集合名
- `DASHSCOPE_API_KEY`：千问平台密钥
- `DASHSCOPE_BASE_URL`：兼容 OpenAI 的 base URL
- `EMBEDDING_MODEL`：嵌入模型名（默认 `text-embedding-v4`）

### 2.3 加载并切分知识库文本

调用 `load_and_chunk(data_dir="../data")`（来自 `ingest/loader.py`）：

1. 遍历 `data` 目录下 `.txt/.md` 文件
2. 读取文本内容
3. 使用 `RecursiveCharacterTextSplitter` 切分
   - `chunk_size=800`
   - `chunk_overlap=120`
4. 为每个 chunk 生成元数据：
   - `source`（来源文件名）
   - `chunk_id`（块编号）

函数最终返回：

- `[(chunk_text, metadata), ...]`

### 2.4 文本向量化（千问 embedding）

创建 `DashScopeEmbeddings`（`app/embeddings.py`）：

- 内部使用 `OpenAI(api_key=..., base_url=...)`
- 调用 `client.embeddings.create(...)`

这个类实现了 LangChain 的 `Embeddings` 接口：

- `embed_documents(texts)`：给多段文本生成向量
- `embed_query(text)`：给单个查询生成向量

你之前碰到的 400 报错，就是这里通过自定义适配器修复的。

### 2.5 写入 Qdrant

调用：

- `QdrantVectorStore.from_texts(...)`

传入：

- 文本列表 `texts`
- 元数据 `metadatas`
- embedding 适配器
- Qdrant 连接参数
- collection 名

执行完成后，Qdrant 中就有向量数据可检索。

## 3. 再看在线服务启动流程（`app/main.py`）

启动方式：

```bash
python app/main.py
```
或
```bash
uvicorn app.main:app --reload --port 8000
```

### 3.1 模块导入与全局初始化

`main.py` 被加载时，会依次执行：

1. `app = FastAPI(title="Helpdesk AI")`
2. `router = build_router(QDRANT_COLLECTION)`

注意这一步非常关键：`build_router(...)` 会提前构建好链路对象。

### 3.2 build_router 做了什么（`app/chains.py`）

`build_router(collection_name)` 内部做两件事：

1. `build_intent_chain(...)`：构建“意图识别链”
2. `build_rag_chain(...)`：构建“FAQ 检索回答链”

最后返回一个 `RunnableLambda(route)`，即一个“可调用路由器函数”。

## 4. 详细看一次 `/chat` 请求如何执行

接口定义在 `main.py`：

- `@app.post("/chat", response_model=ChatResponse)`

请求体模型是 `ChatRequest`（`router_schemas.py`）：

- `session_id`
- `user_id`
- `query`

假设你发来：

- `query = "你们服务时间是什么？"`

执行流程如下。

### 第 1 步：记录用户消息

`append_turn(session_id, "user", query, {"user_id": ...})`

这会写入 `memory.py` 中 `_sessions`（进程内字典）。

### 第 2 步：调用路由器

`routed = router.invoke({"query": req.query})`

这里的 `router` 就是 `build_router` 返回的那个 `RunnableLambda(route)`。

### 第 3 步：意图识别（intent chain）

在 `route(payload)` 里先执行：

- `intent_chain.invoke({"query": ..., "labels": [...]})`

`intent_chain` 结构是：

- `INTENT_PROMPT -> ChatOpenAI(qwen) -> StrOutputParser`

意思：

1. 用提示词要求模型从 `FAQ/TICKET/COMPLAINT/ESCALATION` 中选意图
2. 返回 JSON 文本
3. 解析文本

当前代码里会 `json.loads(...)`，解析失败就默认 `FAQ`。

### 第 4 步：分支处理

#### 分支 A：FAQ

如果意图是 `FAQ`，执行 `rag_chain.invoke({"query": ...})`。

`rag_chain` 的结构：

1. `RunnablePassthrough()` 透传输入
2. `RunnableLambda(fetch_ctx)` 检索上下文
3. `RAG_PROMPT` 组装最终问答提示词
4. `ChatOpenAI(qwen)` 生成答案
5. `StrOutputParser()` 输出字符串

其中 `fetch_ctx` 做的事：

1. `retriever.get_relevant_documents(query)` 从 Qdrant 检索 top-k（k=4）
2. 把每个文档片段拼成可读引用文本（含 source/chunk_id）
3. 返回 `{query, contexts}` 供后续 prompt 使用

最终 `route` 返回：

- `{"intent": "FAQ", "answer": "...", "actions": []}`

#### 分支 B：TICKET / COMPLAINT / ESCALATION

`route` 不直接给答案，而是返回动作：

- `actions = ["create_or_update_ticket"]`

回到 `main.py` 后进入工单逻辑。

### 第 5 步：工单逻辑（main.py + tools.py）

当 `actions` 包含 `create_or_update_ticket` 时：

1. 从 slots 里取 `issue/priority`（没有就兜底）
2. 调 `create_or_update_ticket(...)`
3. 该函数在 `_FAKE_DB` 里生成一个工单记录（当前为内存模拟）
4. 返回 `ticket_id` 和状态
5. 拼接响应文案并写入会话记忆

最终返回 `ChatResponse(intent, answer, ticket_id)`。

### 第 6 步：普通 FAQ 响应

如果没有工单动作：

1. 用 `routed["answer"]` 作为回复
2. 记忆里写一条 assistant turn
3. 返回 `ChatResponse(intent, answer)`

## 5. `/feedback` 接口执行流程

`/feedback` 更简单：

1. 接收 `FeedbackRequest(session_id, score, comment)`
2. `append_turn(..., role="feedback", text="score=..., comment=...")`
3. 返回 `{ "ok": true }`

它只是记录反馈，不会触发模型调用。

## 6. 核心组件关系图（文字版）

1. `main.py` 是“交通枢纽”
2. `chains.py` 是“智能决策中枢”
3. `loader.py + index_qdrant.py` 是“知识准备流水线”
4. `embeddings.py` 是“千问向量化适配层”
5. `memory.py/tools.py` 是“会话和工单状态层（当前内存版）`

可以记一个口诀：

- 先建库（ingest）
- 再启动（app）
- 聊天先路由（intent）
- FAQ走检索（RAG）
- 复杂诉求走工单（action）

## 7. 你需要特别理解的 3 个关键点

### 关键点 1：RAG 不等于“直接问模型”

它是“先检索，再回答”。

- 不检索：模型容易胡编
- 先检索：回答更贴近你的知识库

### 关键点 2：`router` 是“可执行链”对象

`router.invoke(...)` 背后不是单个函数，而是 LangChain 组合出来的可运行图。

### 关键点 3：当前记忆与工单都是“内存态”

重启进程后会丢失：

- `_sessions` 会清空
- `_FAKE_DB` 会清空

生产环境要替换成数据库（如 Redis/MySQL/PostgreSQL）。

## 8. 初学者调试建议（按顺序）

1. 先跑 `python ingest/index_qdrant.py`，确认向量入库成功。
2. 再启动 `python app/main.py`。
3. 用 curl/Postman 调 `/chat`，先测 FAQ，再测工单。
4. 如果结果异常，优先打印：
   - `intent_json`（看意图链输出）
   - `docs`（看检索是否命中）
   - `routed`（看最终路由结果）

## 9. 目前代码里你后续可优化的方向

1. `chains.py` 里的中文提示词有乱码，建议改成 UTF-8 正常中文。
2. `tools.py`/`memory.py` 从内存切到持久化存储。
3. 给 Qdrant 使用 HTTPS，避免 `api key + http` 告警。
4. 给 `route` 增加更严格的 JSON 结构校验（防止模型输出脏格式）。
5. 在 RAG 返回里增加“置信度/命中文档数”元信息，便于前端展示。
